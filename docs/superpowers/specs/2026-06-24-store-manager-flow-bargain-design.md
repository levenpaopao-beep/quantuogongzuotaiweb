# 店长工作台与议价申请设计

## 背景

当前店长入口存在两个核心问题：

1. 店长找不到在线填写议价申请的位置，现有系统更偏向上传 Temu 议价申报表后生成回复。
2. 首页同时出现“工作流程、操作节奏、开始使用清单、今日销量填报进度、今日待处理”等模块，内容重复，店长不清楚每天进入系统后应该先做什么。

本次调整目标是把店长日常工作收敛为高频、清晰、可执行的入口；同时新增在线议价申请模块，支持店长输入一个商家编码后自动拉取同货品下所有尺码数据。

## 首页模块收敛

首页不再重复展示多个“流程说明”模块，只保留一个主工作区：

1. 销量补录
2. 议价申请
3. 任务包处理
4. 导入/同步缺口提醒
5. 经营结果查看

“今日销量”文案改为“销量补录”或“待填销售日”。原因是店长实际填写的通常不是当天销量，而是晚 1-2 天补录的销售日期。系统默认优先提示 T-1、T-2 或最近缺失销售日，不再用“今天必须填今日销量”的表达。

角色入口仍然是同一个工作台：

- 店长只看到自己负责的平台、店铺、任务和销量缺口。
- 管理员看到全部平台、店铺、缺口、任务包和确认队列。

## 店长日常流程

店长每天进入系统后的操作顺序：

1. 先看“待填销售日”，补齐自己负责店铺的销量。
2. 如果有议价需求，进入“议价申请”，在线填写商家编码。
3. 处理管理员推送的任务包，按整包提交。
4. 查看是否有导入/同步缺口提醒，例如某店铺缺少周度销售导入或 ERP 同步数据。
5. 查看经营结果和异常提示。

首页只呈现真实待办，不再重复展示使用说明。新手引导可以保留为折叠帮助或空状态提示，不占用日常首页主区域。

“议价申请”入口需要常驻店长首页主动作区。即使店长当前没有任务包，也可以主动发起议价申请，避免店长找不到填报入口。

## 议价申请录入

议价申请必须是在线表单，不走上传表作为店长主流程。

店长发起议价时，先确认本次议价申请所属店铺，再输入一个商家编码并按回车：

1. 系统记录本次议价申请的店铺。
2. 系统用商家编码反查对应货品编码。
3. 系统找到同一货品编码下所有尺码。
4. 系统把这些尺码的平台销售、申报价、备货和库存数据生成到暂存区。
5. 暂存区按尺码维度展示，一行就是一个尺码/商家编码的一次议价。

如果店长只负责一个店铺，系统默认带出该店铺；如果店长负责多个店铺，必须先选择店铺后再录入商家编码。一个店铺只属于一个平台，所以店铺选定后平台自动确定。

Temu 和 Shein 共用同一套议价申请、审批、历史和低价回追流程。平台差异只放在数据字段映射层：Temu 使用 Temu 平台表里的申报价、30 天销量和链接/SKC；Shein 使用 Shein 平台表里的申报价、30 天销量和链接/SKC。

店长一次可以提交多个货品。交互上，店长输入一个商家编码并回车后加入暂存区，可以继续输入第二个商家编码追加到同一个暂存区，最后一次性提交。系统按 `店铺 + 平台 + 提交人 + 提交时间` 生成一个议价申请批次；批次下可以包含多个货品编码和多条尺码议价行。

暂存区主表字段：

- 货品名称
- 议价申请店铺
- 卖得最好的店铺
- 尺码
- 商家编码
- 本次议价
- 成本价
- 批发价
- 建议申报价/批发价占比
- 在线销售链接数
- 在售最低申报价
- Temu 30 天最高销量
- Shein 30 天最高销量
- 平台仓库备货库存
- ERP 返回库存

货品名称必须放在最前面，让店长先知道当前是什么产品。议价申请店铺和卖得最好的店铺也必须在主表可见，因为它们直接影响这次议价是否合理。

“卖得最好的店铺”按货品维度计算，不按单个尺码计算。系统用同一货品编码下全部尺码的最近 30 天销量，跨平台、跨店铺汇总后取销量第一的店铺。暂存区每个尺码行展示同一个货品级“卖得最好的店铺”，用于帮助店长判断这个货品在哪个店铺表现最好。

如果多个店铺并列第一，主表显示“店铺A / 店铺B 并列”，详情里展示各自销量。系统不强行选择其中一个店铺。

“在线销售链接数”在暂存区主表按当前尺码/商家编码计算，因为一行就是一个尺码的议价。商品识别区顶部另展示“货品总在线链接数”，按同一货品编码下全部尺码的在线链接汇总计算。

议价暂存区必须显示 ERP 返回的成本价、批发价，以及建议申报价占批发价的比例。成本价是能否同意议价的底线：除非该货品所属分类已经被标记为清仓类，否则建议申报价低于成本价时必须标红，并提示不建议通过。批发价占比用于辅助判断平台要求价格是否过低，例如“建议申报价为批发价的 82%”。非清仓货品低于成本价时仍允许店长提交，但必须强红提示，并在管理员审批页默认标为“建议不通过”。

清仓判断第一版不维护分类 ID 白名单，而是从当前导入或同步的 ERP 源文件里生成“清仓款式基础表”：

1. 读取 ERP 商品基础信息里的 `货品分类` 字段。
2. `货品分类` 包含“清仓”的货品进入清仓款式基础表。
3. 清仓款式基础表字段包括：`货品编码`、`货品名称`、`商家编码`、`尺码`、`规格名称`、`清仓分类`。
4. 议价判断时按 `货品编码` 命中整款；只要一个货品编码在清仓款式基础表中，该货品编码下全部尺码都按清仓逻辑处理。
5. 后续更新方式支持两种：管理员导入 ERP 商品基础表更新，或 ERP 接口拉取分类后自动更新。

清仓货品的建议申报价低于成本价时，不标红硬拦截，改为橙色提示：“清仓款，允许低于成本价，请确认是否以清库存为目标”。非清仓货品低于成本价仍标红并提示不建议通过。

清仓款式基础表入口放在“基础数据”模块下，名称为“清仓款式”。该入口仅管理员维护，支持从 ERP 商品基础表一键重建、查看/搜索清仓货品、导出清仓款式表。店长不维护该表，只在议价暂存区看到清仓提示。

同款核价风险按平台内同货品编码计算。系统需要识别同一个平台、同一个货品编码下，不同店铺和不同尺码之间的低价锚点。如果某个店铺或某些尺码已经出现明显低价，导致平台后续可能要求其他店铺/尺码跟价，暂存区需要给出风险等级：

- 绿色：当前建议申报价接近同款最低价，低价锚点风险小。
- 黄色：当前建议申报价高于同款最低价，但价差不大，需要关注。
- 红色：同款跨店铺/跨尺码已经出现明显低价，当前建议申报价大概率会被平台议价压低，需要管理员重点确认。

## 本次议价默认值

“本次议价”不做空白表。系统先给建议值，店长可以逐行修改。

默认建议逻辑：

1. 找到当前尺码的在售最低申报价。
2. 按可配置降价幅度生成建议议价。
3. 如果缺少申报价数据，则本次议价为空，并提示店长手动填写。

该建议只辅助填写，不做强制拦截。

## 明细与审核

主表按尺码汇总，适合店长日常录入。Temu、Shein、平台仓、ERP 的来源明细默认收起，点击某一行后展开。

店长提交后不直接生成平台回复，而是进入管理员确认队列。管理员不修改店长填写的议价，只能选择通过或不通过。管理员按行审批，一个申请批次下的不同货品/尺码可以有不同审批结果。审批页支持多选后一键通过或一键不通过，最后点击完成审批。管理员审批时可以填写简短备注，备注不是必填。通过后流程结束并归档，不要求店长再回复“已执行”；不通过时由店长重新提高议价后再提交。

不通过的议价行需要回到店长的“需调整议价”列表，并关联原申请记录。店长重新提交时只能修改本次议价，系统生成新版本，例如“第 2 次提交”。议价历史里需要展示同一商家编码的多次提交价格、审批结果和审批备注，方便追溯调整过程。

系统需要保留历史议价操作结果，并提供“议价历史”查询入口。管理员可以按货品编码、商家编码、店铺、平台、店长、审批结果、时间范围查询历史记录，看到店长提交价、管理员审批结果、审批备注、审批时间。该入口用于回追低价来源：判断某个低价是管理员审批通过过，还是店长未报备自行上线低价。

“议价历史”入口放在管理员视角下，分为两个标签：`审批记录` 和 `低价回追`。审批记录用于查询店长提交与管理员审批结果；低价回追用于检查当前平台在线低价是否存在已通过审批记录，并支持多选一键忽略历史低价。

后续当平台表或销售数据里发现低于历史审批价、低于成本价、低于同款风险价的在线价格时，系统应能提示“未匹配到已通过议价记录”或“存在历史审批记录”，帮助管理员追溯责任。

由于系统上线前已经存在历史低价链接，低价回追必须支持忽略机制。管理员可以把某条低价风险标记为“已知低价/忽略”，忽略按单个商家编码生效。忽略记录保存当时的申报价作为“忽略底价”；后续同一平台、店铺、商家编码的当前价格不低于忽略底价时不再重复提示，如果价格继续下降、店铺变化或商家编码变化，则重新提示。忽略记录需要保留操作人、时间和备注，方便后续追溯。

低价回追列表支持多选一键忽略。管理员多选风险后点击“一键忽略”，系统弹出确认框，显示本次忽略条数，并允许填写统一备注，例如“上线前历史低价”。确认后，系统按每条记录各自的平台、店铺、商家编码和当前申报价生成忽略记录。

第一版低价回追只覆盖 Temu 和 Shein。其他平台保留平台字段和扩展位置，但不进入第一版低价回追规则。低价回追在每次导入或同步 Temu、Shein 平台销售/申报价数据后自动刷新。管理员也可以在 `议价历史 > 低价回追` 手动点击“重新检查”。系统不做脱离数据更新节奏的每日强制检查，避免用旧数据重复提醒。

低价回追比较历史审批价时需要保留误差容忍。默认容忍值为 `0.05 元`：当前在线申报价不低于“历史审批价 - 0.05”时不提示；低于该值才提示低于审批价。该容忍值放在规则设置里，管理员可调整。

## 异常处理

议价判断条件作为提醒，不硬拦截：

- 在线链接数过少或为 0
- 在售最低申报价缺失
- Temu 或 Shein 30 天销量异常
- 平台仓库备货库存不足
- ERP 返回库存不足或缺失
- 平台销售数据与店长填写/导入数据存在明显差异

这些异常在暂存区行内提示，管理员确认时也能看到。

## 数据来源

第一版优先使用现有本地数据和手动同步数据：

- ERP 商品信息
- ERP 库存信息
- Temu 平台销售/申报价数据
- Shein 平台销售/申报价数据
- 平台仓库备货库存
- 店长手动填写的销售件数

ERP 自动同步作为可配置能力保留，第一版以手动同步为主，自动同步开关后续再完善。

## 验收标准

1. 店长首页不再出现多个重复流程模块。
2. 店长能在首页明确看到“销量补录、议价申请、任务包处理”三个主动作。
3. 销量模块不再以“今日销量”为核心文案，而是以“销售日期/待填销售日”为核心。
4. 店长可以从明确入口进入议价申请。
5. 输入一个商家编码后，系统能反查货品编码，并生成同货品编码下所有尺码的暂存行。
6. 暂存区一行代表一个尺码/商家编码的议价。
7. 暂存区最前面显示货品名称。
8. 暂存区主表显示议价申请店铺和卖得最好的店铺。
9. 本次议价有建议值，且允许店长逐行修改。
10. 店长提交后进入管理员确认队列，不直接生成最终回复。
11. 管理员确认后，该申请从待办中消失。
